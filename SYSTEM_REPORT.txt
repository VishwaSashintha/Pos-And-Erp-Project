 CHAPTER 1: INTRODUCTION
================================================================================

1.1 BACKGROUND OF THE SYSTEM
------------------------------
Gradge ERP is a cloud-based, multi-tenant Enterprise Resource Planning (ERP)
system designed specifically for the automotive garage and small-to-medium
business (SMB) sector. Traditional ERP solutions such as SAP and Oracle are
prohibitively expensive and overly complex for small businesses like garages,
workshops, and retail stores. Gradge ERP was built to bridge this gap by
providing a lightweight, modular, and affordable SaaS alternative.

The system is built on a modern Java Spring Boot backend with a React + TypeScript
frontend, deployed using Docker, and follows a shared-database multi-tenancy model
where each business (tenant) operates in complete isolation from others using a
tenant_id column strategy across all data tables.

1.2 PROBLEM STATEMENT
----------------------
Small and medium-sized garage businesses face significant operational challenges:

  - Manual bookkeeping leads to financial errors and lack of visibility
  - No integrated system to track inventory, job cards, and customer vehicles
  - Difficulty managing employees, roles, and work assignments
  - No real-time insight into sales, purchases, or profit/loss
  - Existing ERP tools are too expensive or too complex to implement
  - No system to handle multi-location or multi-branch stock management

These inefficiencies result in revenue leakage, poor customer service, and
inability to scale the business.

1.3 OBJECTIVES
--------------
The primary objectives of the Gradge ERP system are:

  1. To design and implement a fully functional multi-tenant SaaS ERP system
     for the automotive and general SMB sector.

  2. To provide a modular system covering: Point of Sale (POS), Inventory
     Management, Financial Accounting, Workshop Management, CRM, HR & Payroll,
     Purchase Orders, Reporting, and Billing.

  3. To implement a robust, secure authentication system with role-based access
     control (RBAC) and fine-grained permissions.

  4. To ensure complete data isolation between tenants using a shared-database
     multi-tenancy architecture.

  5. To provide real-time notifications, audit logging, and event-driven
     communication between modules using RabbitMQ.

  6. To build a responsive web frontend using React and TypeScript that allows
     businesses to manage all operations from a single dashboard.

  7. To deliver a system that is production-ready with Docker containerization,
     CI/CD pipelines, and cloud deployment capability.

1.4 SCOPE OF THE SYSTEM
------------------------
The Gradge ERP system covers the following functional areas:

  INCLUDED IN SCOPE:
  - Multi-tenant SaaS architecture with tenant onboarding
  - User registration, login, logout, and password reset
  - Role-based access control (8 roles, 10 permissions)
  - Point of Sale (POS) with invoice creation and confirmation
  - Inventory management with multi-warehouse stock tracking
  - Financial accounting with double-entry bookkeeping (ledger, journal, trial
    balance, balance sheet, profit & loss)
  - Purchase Order (PO) management with goods receiving
  - Customer Relationship Management (CRM)
  - Workshop module with job cards and vehicle inspection
  - Vehicle management (customer vehicle registry)
  - Employee management and HR onboarding flows
  - Supplier management
  - Audit logging for critical operations
  - Real-time system events via RabbitMQ
  - WebSocket support for live dashboard updates
  - File uploads via MinIO (S3-compatible object storage)
  - Redis-backed JWT blocklist and distributed rate limiting
  - Spring Cache for performance optimization
  - Reporting and data export (PDF, Excel)
  - Subscription/billing module with plan-based feature gating
  - Docker-based deployment with docker-compose
  - CI/CD pipeline via GitHub Actions

  OUT OF SCOPE:
  - Native mobile applications (iOS / Android)
  - Third-party payment gateway integration (Stripe/PayPal) - planned Phase 3
  - OAuth2 social login (Google/GitHub) - planned Phase 3
  - Payroll processing with tax computation
  - Advanced analytics / ML-powered forecasting

1.5 LIMITATIONS
---------------
  - The current system uses a shared PostgreSQL database with tenant_id-based
    isolation, which means a misconfigured query could theoretically expose
    cross-tenant data (mitigated by TenantIsolationInterceptor).

  - The JWT secret key is stored in application.yml — in production this must
    be migrated to a secret management service (e.g., AWS Secrets Manager).

  - The email system currently connects to a local MailHog SMTP instance for
    development. Real production email requires an SMTP relay configuration.

  - Redis and RabbitMQ are single-node instances. High availability clustering
    is required for production workloads.

  - The frontend React application is a functional prototype. Full mobile
    responsiveness and UX polish are planned for future releases.

  - The system currently does not support localization (multi-language/i18n).

2.1 CURRENT SOLUTIONS IN THE MARKET
-------------------------------------
Several ERP solutions exist for businesses, ranging from large enterprise tools
to lightweight open-source alternatives:

  a) SAP Business One
     - Enterprise-grade ERP widely used in manufacturing and distribution.
     - Very high licensing cost ($1,500–$3,000/user/year).
     - Requires dedicated IT teams to manage and configure.
     - Too complex for small garage businesses.

  b) Odoo (Community & Enterprise)
     - Open-source ERP with extensive module support.
     - Community edition is free but limited; Enterprise costs ~$24/user/month.
     - Highly customizable but requires technical expertise.
     - Used widely in SMBs but has a steep learning curve.

  c) Zoho Books / Zoho One
     - Cloud-based SaaS accounting and business suite.
     - Affordable pricing ($12–$37/month).
     - Limited workshop or vehicle management features.
     - Not specifically designed for garages or automotive sector.

  d) QuickBooks Online
     - Popular accounting SaaS tool focused on invoicing and bookkeeping.
     - No inventory, job card, or workshop management features.
     - Primarily targeted at accountants, not garage operators.

  e) GarageHive / Motivity
     - Automotive-specific workshop management software.
     - Focuses mainly on job cards and vehicle history.
     - Does not include full accounting, inventory, or multi-tenancy.

2.3 IDENTIFIED GAPS
--------------------
Based on the analysis above, the following gaps were identified that Gradge ERP
addresses:

  1. No affordable, automotive-specific ERP system exists with a full SaaS
     multi-tenant architecture.

  2. Existing tools lack integration between workshop job cards, vehicle records,
     inventory, and financial accounting in a single platform.

  3. Real-time event-driven notifications (low stock, invoice confirmed, job
     card completed) are absent from most SMB-targeted tools.

  4. Fine-grained RBAC with 8 roles and named permissions is uncommon in tools
     at this price point.

  5. Most tools require complex setup. Gradge ERP uses a simple tenant
     registration flow that auto-provisions all required modules.

3.1 REQUIREMENTS GATHERING

Requirements were gathered through:

3.2 FUNCTIONAL REQUIREMENTS
-----------------------------
FR-01: The system shall allow a new business to register as a tenant with a
        unique name, admin user, and selected modules.

FR-02: The system shall support user login with JWT-based authentication and
        provide a refresh token mechanism.

FR-03: Users shall be able to reset their password via a time-limited email token.

FR-04: The system shall support 8 user roles, each with a defined set of
        permissions, enforced via Spring Security @PreAuthorize.

FR-05: The system shall allow creation, viewing, and confirmation of POS invoices
        with automatic ledger journal entry creation.

FR-06: The system shall manage inventory stock per product, per warehouse, and
        per tenant, with stock movement history.

FR-07: The system shall raise a low-stock alert event when stock falls at or
        below the configured reorder level.

FR-08: The system shall support double-entry bookkeeping with journal entries,
        chart of accounts, trial balance, and balance sheet generation.

FR-09: The system shall manage purchase orders from creation through approval
        to goods receipt, with automatic stock updates.

FR-10: The system shall support job card creation, technician assignment, and
        job card completion in the workshop module.

FR-11: The system shall maintain a vehicle registry linked to customers and
        job cards.

FR-12: The system shall log all critical operations (e.g., tenant registration,
        invoice creation) in an audit log with user and timestamp.

FR-13: The system shall support file uploads (PDFs, images) via MinIO S3.

FR-14: The system shall provide a subscription/billing module with plan-based
        feature gating using FeatureGuard.

FR-15: The system shall expose a REST API documented via Swagger/OpenAPI.

3.3 NON-FUNCTIONAL REQUIREMENTS
---------------------------------
NFR-01  SECURITY       - All API endpoints (except /api/auth/**) must be
                         authenticated via JWT. Passwords stored with BCrypt.

NFR-02  PERFORMANCE    - Frequently queried data (chart of accounts, feature
                         access) must be cached using Spring Cache with Redis.

NFR-03  SCALABILITY    - The system must support multiple tenants sharing the
                         same database without performance degradation.

NFR-04  RATE LIMITING  - API requests must be rate-limited at 100 req/minute
                         per client, enforced via Redis atomic counters.

NFR-05  AVAILABILITY   - The system must be containerized (Docker) to support
                         zero-downtime deployment.

NFR-06  AUDITABILITY   - All critical actions must be logged in the audit_logs
                         table with user, action, and timestamp metadata.

NFR-07  MAINTAINABILITY- Code must follow a layered architecture (Controller
                         -> Service -> Repository) with AOP for cross-cutting
                         concerns.

NFR-08  TESTABILITY    - Core business flows must have integration tests covering
                         happy path and edge cases.

NFR-09  API STANDARDS  - All REST responses must follow a unified ApiResponse<T>
                         wrapper with success/failure status and message.

NFR-10  EXTENSIBILITY  - New modules must be addable without modifying the core
                         tenant or security infrastructure.

4.1 ARCHITECTURE OVERVIEW (SaaS MULTI-TENANT ARCHITECTURE)
-----------------------------------------------------------
Gradge ERP uses a SHARED DATABASE, SEPARATE TENANT_ID architecture:

  - All tenants share one PostgreSQL database.
  - Every table that holds tenant-specific data includes a "tenant_id" (UUID)
    column which acts as the partition key.
  - A TenantContext (ThreadLocal) stores the current tenant's UUID for the
    duration of each request.
  - A TenantFilter (Servlet filter) reads the X-Tenant-ID header from each
    request and populates TenantContext.
  - A TenantIsolationInterceptor validates that the JWT's tenant claim matches
    the request's tenant ID, preventing cross-tenant data access.

4.2 SYSTEM ARCHITECTURE DIAGRAM
---------------------------------
(Refer to diagram above in section 4.1 for the full system architecture)

Key architectural patterns used:
  - Layered Architecture: Controller -> Service -> Repository
  - AOP (Aspect-Oriented Programming): @Auditable annotation intercepts
    methods and writes to audit_logs table automatically
  - Event-Driven Architecture: EventPublisherService publishes domain events
    (STOCK_LOW, INVOICE_CONFIRMED, JOB_CARD_COMPLETED) via RabbitMQ/AMQP
  - Domain-Driven Design (DDD): Workshop and Vehicle modules use use-cases,
    repository ports (interfaces), and infrastructure adapters
  - CQRS Partial: Read-heavy views (reports, dashboard) separated from
    write operations in service layers

4.5 SEQUENCE DIAGRAMS
-----------------------
SEQUENCE: TENANT REGISTRATION
  Client --> AuthController: POST /api/auth/register-tenant
    AuthController --> AuthService: registerTenantAndAdmin(...)
      AuthService --> TenantRepository: save(tenant)
      AuthService --> TenantSubscriptionRepository: save(subscription)
      AuthService --> LedgerService: initializeDefaultAccounts(tenantId)
        LedgerService --> AccountRepository: save(8 default accounts)
      AuthService --> UserRepository: save(adminUser)
      AuthService --> NotificationService: sendEmail(welcome email)
    AuthService --> AuthController: {tenantId, username, message}
  AuthController --> Client: 200 OK ApiResponse

SEQUENCE: POS INVOICE CONFIRMATION
  Client --> InvoiceController: POST /api/invoices/{id}/confirm
    InvoiceController --> InvoiceService: confirmInvoice(id, tenantId)
      InvoiceService --> FeatureGuard: check(tenantId, POS)
        FeatureGuard --> SubscriptionService: hasFeatureAccess(...)
          [cache hit or DB lookup]
      InvoiceService --> InvoiceRepository: findById(id)
      InvoiceService --> LedgerService: recordTransaction(journal entry)
        LedgerService --> AccountRepository: update balances (debit/credit)
        LedgerService --> JournalEntryRepository: save(entry)
      InvoiceService --> StockService: reduceStock(product, qty)
        StockService --> StockRepository: save(updated stock)
        [if stock <= reorderLevel] EventPublisher: publish(STOCK_LOW event)
      InvoiceService --> EventPublisher: publish(INVOICE_CONFIRMED)
      InvoiceService --> NotificationService: sendEmail(invoice confirmation)
    InvoiceController --> Client: 200 OK

SEQUENCE: PASSWORD RESET
  Client --> AuthController: POST /api/auth/forgot-password {email}
    AuthController --> AuthService: forgotPassword(email)
      AuthService --> UserRepository: findByEmail(email)
      AuthService --> PasswordResetTokenRepository: deleteByUserId(userId)
      AuthService --> PasswordResetTokenRepository: save(new token, 1h TTL)
      AuthService --> NotificationService: sendEmail(reset token)
    AuthController --> Client: 200 OK (generic message)

  Client --> AuthController: POST /api/auth/reset-password {token, newPassword}
    AuthController --> AuthService: resetPassword(token, newPassword)
      AuthService --> PasswordResetTokenRepository: findByToken(token)
      [check expiry] AuthService --> UserRepository: save(BCrypt(newPassword))
      AuthService --> PasswordResetTokenRepository: delete(token)
    AuthController --> Client: 200 OK

5.3 KEY FEATURES IMPLEMENTATION
---------------------------------
a) MULTI-TENANCY
   - TenantFilter reads X-Tenant-ID header and sets TenantContext.
   - TenantIsolationInterceptor validates JWT tenant claim matches header.
   - All service methods receive tenantId as a parameter or read from context.

b) DOUBLE-ENTRY ACCOUNTING
   - LedgerService.recordTransaction() validates that total debits == credits.
   - Automatically updates account balances based on account type:
     Assets/Expenses: debit increases, credit decreases
     Liabilities/Equity/Revenue: credit increases, debit decreases
   - Invoice confirmation auto-generates: DR Accounts Receivable / CR Revenue

c) FEATURE GATING
   - FeatureGuard.check(tenantId, FeatureKey) verifies tenant has module active.
   - Results are @Cacheable to avoid repeated DB lookups.
   - Example: POST /api/invoices/** requires FeatureKey.POS to be enabled.

d) RATE LIMITING (Redis)
   - RateLimiterService.isAllowed(key) increments an atomic Redis counter.
   - First request sets a 60-second expiry window.
   - Returns false if count exceeds 100 requests per window.

e) AUDIT LOGGING
   - @Auditable(action = "ACTION_NAME") marks methods to be audited.
   - AuditAspect intercepts these calls via @Around and records:
     action, user, tenantId, timestamp to the audit_logs table.

f) PASSWORD RESET
   - Generates a UUID token, stores in password_reset_tokens with 1h expiry.
   - Sends token via email. On reset, validates expiry then BCrypt-encodes
     new password and invalidates the token.

g) MULTI-WAREHOUSE STOCK
   - Stock entity references both Product and Warehouse.
   - StockService.addStock/reduceStock are overloaded: with/without warehouse.
   - getStock() aggregates across all warehouses for a product.
   - Low-stock check is on total aggregated quantity, not per-warehouse.

5.5 SECURITY IMPLEMENTATION
-----------------------------
a) AUTHENTICATION
   - JWT (JSON Web Token) issued on login via JwtService.
   - Token contains: username, role, tenantId (as claims).
   - Token expiry: 24 hours (86400000 ms), configurable in application.yml.
   - Refresh tokens stored in DB with 7-day expiry.
   - On logout: JWT added to Redis blocklist with TTL = remaining JWT lifetime.

b) AUTHORIZATION
   - Spring Security's @EnableMethodSecurity enables method-level security.
   - @PreAuthorize("hasAuthority('PERMISSION')") guards individual endpoints.
   - JwtAuthenticationFilter validates the token and extracts authorities.
   - User authorities are derived from UserRole.getPermissions() set.

c) PASSWORD SECURITY
   - Passwords stored using BCryptPasswordEncoder (strength factor 10+).
   - Password reset uses UUID token with 1-hour expiry stored in DB.

d) TENANT ISOLATION
   - TenantIsolationInterceptor rejects requests where JWT tenant ≠ header.
   - All repository queries are scoped to tenantId.

e) RATE LIMITING
   - Redis-backed rate limiter enforces 100 requests/60 seconds per client IP.
   - Distributed and stateless - works across multiple server instances.

f) CORS
   - Allowed origins: http://localhost:3000, http://localhost:5173
   - Configured in application.yml, applied via Spring Security CORS config.

g) CSRF
   - CSRF protection disabled (stateless JWT API, no session cookies).

h) SECURE ENDPOINTS
   - /api/auth/** : Public (no authentication required)
   - /swagger-ui/**, /v3/api-docs/** : Public (documentation)
   - /actuator/** : Public (monitoring - should be restricted in production)
   - All other endpoints : Require valid JWT

8.3 PERFORMANCE ANALYSIS
--------------------------
Performance optimizations implemented:

  a) CACHING
     - SubscriptionService.hasFeatureAccess() is @Cacheable("featureAccess")
       -> Eliminates DB query on every feature-gated API call.
     - LedgerService.getChartOfAccounts() is @Cacheable("chartOfAccounts")
       -> Chart of accounts cached per tenant for report generation speed.
     - Cache backend: Redis (spring-boot-starter-data-redis)
     - @EnableCaching enabled on ErpApplication main class.

  b) RATE LIMITING
     - Redis atomic INCR with TTL prevents API abuse.
     - Distributed: works correctly even with multiple backend instances.

  c) DATABASE INDEXES
     - idx_product_tenant: Products indexed by tenant_id
     - idx_warehouse_tenant: Warehouses indexed by tenant_id
     - idx_pr_token_val: Password reset tokens indexed by token value
     - idx_pr_token_tenant: Password reset tokens indexed by tenant_id
     - Compound indexes on frequently queried columns (product_id + tenant_id)

  d) LAZY LOADING
     - JPA FetchType.LAZY on all @ManyToOne and @OneToMany relationships.
     - Prevents N+1 query issues in listing endpoints.

  e) ASYNC PROCESSING
     - @EnableAsync on application entry point.
     - Notification emails are sent asynchronously via Spring's @Async support.
     - RabbitMQ event publishing is fire-and-forget.

B. EXTRA DIAGRAMS
------------------
B.1 Multi-Tenant Data Flow
    Client Request
      --> TenantFilter (sets TenantContext from X-Tenant-ID header)
      --> JwtAuthenticationFilter (validates JWT, extracts user/role/tenantId)
      --> TenantIsolationInterceptor (cross-validates JWT.tenantId == header)
      --> Controller
      --> Service (reads TenantContext.get() for tenant-scoped queries)
      --> Repository (queries filtered by tenantId)
      --> Response

B.2 Event Flow (Stock Low Alert)
    InvoiceService.confirmInvoice()
      --> StockService.reduceStock()
          --> stockRepository.save(updatedStock)
          --> if (stock <= reorderLevel):
              EventPublisherService.publish(SystemEvent{STOCK_LOW})
                --> RabbitMQ exchange (system.events)
                    --> NotificationService listener
                        --> sendEmail(admin, "Low Stock Alert")

B.3 Employee Onboarding Flow
    Admin --> POST /api/employees (creates Employee + inactive User)
      --> EmployeeService generates activation_token (UUID)
      --> NotificationService sends activation email with token
    Employee --> POST /api/auth/onboard {token, password}
      --> AuthController validates token (not expired)
      --> UserRepository.save(BCrypt(password), active=true)
      --> Employee.activation_token = null (invalidated)

    1. Navigate to Sales (POS).
    2. Click "New Invoice".
    3. Select the customer and add products with quantities.
    4. Click "Create Invoice" to save as draft.
    5. Click "Confirm Invoice" to post to accounting (creates journal entry
       and reduces stock automatically).
    6. Print or email the invoice to the customer.

C.5 MANAGING INVENTORY
    1. Navigate to Products to add/edit products with pricing.
    2. Set a Reorder Level on each product to get low-stock alerts.
    3. Receive stock through Purchase Orders (Purchases section).
    4. View stock levels and movement history in the Stock section.

C.6 WORKSHOP OPERATIONS
    1. Register customer vehicles in the Vehicles section.
    2. Create a Job Card linked to a vehicle and customer.
    3. Assign a technician and add inspection items.
    4. When work is done, mark the Job Card as Completed.
    5. Generate an invoice from the completed job card.

C.7 VIEWING FINANCIAL REPORTS
    1. Navigate to Accounting.
    2. View the Chart of Accounts to see all account balances.
    3. View Trial Balance to verify books are balanced.
    4. View Balance Sheet for assets, liabilities, and equity overview.
